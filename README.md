# minnows

Small, authored **agent-ecosystem assets** — CLI tools (optionally skill-wrapped) and
**versioned data packs** — so one-off helpers stop scattering across dotfiles,
`~/.claude/skills`, and ad-hoc repos.

## Product kinds

### 1. Tool minnows — CLI-first, skill-wrapped. Not MCP.

- Every **tool** is a **CLI executable** — the engine. Usable by a human in a terminal
  *and* by an agent shelling out.
- A tool *may* also ship a **`SKILL.md`** — thin wrapper for agent discovery. Skills are
  **opt-in** (CLI-only tools cost zero skill-index budget).
- **MCP is off the table** by default for this class of local tool.

### 2. Data packs — versioned JSON, not CLIs

- Live under **`data/<pack-name>/`** with a required **`pack.json`**.
- No CLI required. No skill by default.
- Discover via **[data/README.md](data/README.md)** (clickable latest / releases) and
  **[data/index.json](data/index.json)** (machine index of latest tags + full URLs).
- Grab: `./scripts/fetch-data-pack.sh <pack>` or the **Download** button on a
  [GitHub Release](https://github.com/tnunamak/minnows/releases?q=data-).

See [data/README.md](data/README.md) for the full pack convention.

## Layout

```
minnows/
├── lib/                       # shared, stdlib-only libraries (dev-time DRY)
├── tools/<name>/              # tool minnow source (CLI ± optional SKILL.md)
├── skills/<name>/             # GENERATED by sync.sh — self-contained shippable skills
├── data/                      # versioned data packs (not CLIs)
│   ├── index.json             #   latest tags + ready-made URLs per pack
│   ├── README.md              #   human/agent discovery table
│   └── <pack>/pack.json       #   pack identity + file list
├── scripts/
│   ├── fetch-data-pack.sh     #   download a pack (latest or version)
│   └── release-data-pack.sh   #   tarball + optional gh release
├── sync.sh
└── install.sh                 # skills + bins + data → ~/.local/share/minnows-data
```

## The packaging rule (why `skills/` is generated)

A skill folder gets copied/symlinked **out** of this repo. At that point `from lib import …`
has no repo-root `lib/` — *on another machine, where you can't see it*.

- **Dev time → shared `lib/` (DRY).**
- **Ship time → vendored / self-contained.** `sync.sh` copies the exe + `lib/` into each
  `skills/*/scripts/`.

Never symlink `lib/` into a skill. Data packs are **read as files** (path / env), not imported
like `lib/` — tools that need a pack resolve `$DATA_PACKS_HOME/<pack>` or a pinned tarball.

## Install

```bash
./install.sh          # sync skills, bins on PATH, data packs → ~/.local/share/minnows-data
```

Idempotent. Dotfiles' `setup.sh` can clone-or-pull and run this. `~/.local/bin` must be on
`$PATH`. After install:

```bash
export DATA_PACKS_HOME="${DATA_PACKS_HOME:-$HOME/.local/share/minnows-data}"
# e.g. $DATA_PACKS_HOME/model-catalog/pack.json
```

## Adding a tool minnow

1. `tools/<name>/<name>` — the CLI (stdlib-only preferred).
2. Optional `tools/<name>/SKILL.md` if agents should auto-discover it.
3. Reuse `lib/` with the existing `_load_lib()` bootstrap pattern.
4. `./install.sh`.

## Adding a data pack

1. `data/<pack>/pack.json` + files + `README.md` with a **Get this pack** table (real links).
2. Add a row to `data/README.md` and an entry in `data/index.json`.
3. `./scripts/release-data-pack.sh <pack> <semver>` then commit + `--push` when ready.
