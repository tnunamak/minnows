# minnows

Small, authored agent tools — each a little fish that swims in the coding-agent ecosystem.
A home (and shared packaging) for the one-off CLIs I keep building for Claude Code / Codex /
Gemini CLI, so they stop scattering across dotfiles, `~/.claude/skills`, and ad-hoc repos.

## What a minnow is

**CLI-first, skill-wrapped. Not MCP.**

- Every minnow is a **CLI executable** — the engine. Usable by a human in a terminal *and* by
  an agent shelling out. This is the substance.
- A minnow *may* also ship a **`SKILL.md`** — a thin wrapper that teaches an agent *when* to
  reach for the CLI and *how* to call it. Skills are **opt-in**: add one only when an agent
  should auto-discover the tool. CLI-only utility minnows cost zero skill-index budget.
- **MCP is deliberately off the table** for this class of small local tool — a CLI behind a
  skill is lighter to build/version/run and strictly more flexible (you can pipe and grep it).
  A future minnow could choose MCP if it genuinely needs server semantics; it's not the default.

## Layout

```
minnows/
├── lib/                       # shared, stdlib-only libraries (dev-time DRY)
│   └── claude_sessions.py     #   locate projects/sessions, parse compaction boundaries
├── tools/<name>/              # the dev source for each minnow
│   ├── <name>                 #   the CLI executable (imports lib/ in dev)
│   ├── SKILL.md               #   OPTIONAL — present iff the tool ships a skill
│   ├── README.md              #   optional human doc
│   └── evals/                 #   optional skill-behavior tests
├── skills/<name>/             # GENERATED by sync.sh — self-contained, shippable
│   ├── SKILL.md
│   └── scripts/{<name>, lib/} #   the exe + a VENDORED copy of lib/
├── sync.sh                    # build skills/ from tools/ (vendor lib into each)
└── install.sh                 # sync + symlink skills into agents + bins onto PATH
```

## The packaging rule (why `skills/` is generated)

A skill folder gets copied/symlinked **out** of this repo (into `~/.claude/skills/`, a
marketplace, another machine). At that point `from lib import …` has no repo-root `lib/` on
the path and dies with `ModuleNotFoundError` — *on another machine, where you can't see it*.

So minnows uses **two patterns by layer** (the 2026 prior-art consensus — Anthropic's own
skills duplicate shared code; the Agent Skills spec requires scripts be self-contained;
PEP 723 explicitly doesn't do local imports):

- **Dev time → shared `lib/` (DRY).** Tools import one source of truth. Edit `lib/` once.
- **Ship time → vendored / self-contained.** `sync.sh` copies the exe + `lib/` into each
  `skills/<name>/scripts/`. Each tool bootstraps its lib from `./lib/` (beside the script)
  or `../../lib/` (repo root), so it runs self-contained anywhere.

Never symlink `lib/` into a skill (breaks on copy-installs + Windows). Never promote `lib/`
to an installed package until an *external* consumer needs versioning — pure ceremony otherwise.

| **convo** | ✓ | ✓ | Read past agent conversation history across Claude Code / Codex / Gemini. Signature view: each user prompt + the agent's final reply, tool noise stripped. Cross-harness search. |
| **uncompact** | ✓ | ✓ | Recover a Claude Code session lost to context compaction — writes a new resumable session truncated to before a chosen compaction boundary. *Claude-only by nature, not omission:* Codex rollouts are append-only (no on-disk history loss) and Gemini stores a full `messages[]` array; both ship native resume/fork. ## Install

```bash
./install.sh          # sync + wire into ~/.{claude,codex,gemini}/skills and ~/.local/bin
```

Idempotent. My dotfiles' `setup.sh` clones-or-pulls this repo and runs `install.sh`
automatically, so a fresh machine gets the tools without extra steps. `~/.local/bin` must be
on `$PATH` (dotfiles shell config handles that).

## Adding a minnow

1. `tools/<name>/<name>` — the CLI (stdlib-only keeps it zero-setup; add PEP 723 `# /// script`
   deps + a `uv run` shebang only if it needs third-party packages).
2. Optional `tools/<name>/SKILL.md` if an agent should auto-discover it.
3. Reuse `lib/` for anything shared; bootstrap it with the `_load_lib()` pattern the existing
   tools use (finds `./lib/` when vendored, `../../lib/` in the repo).
4. `./install.sh`.
